# 爆文改写器 (Viral Article Rewriter)

将普通技术文章改写成具有高传播度的爆文风格，参考《Cursor慌了》《震撼！这个程序员2个月没写一行代码》等成功案例的写作手法。

## 核心理念

爆文不是标题党，而是通过**戏剧性开头 + 数据密集 + 真实案例 + 冲突制造**，在保留专业深度的同时提升传播力。

## 参考标准

**主要参考案例**：《Cursor慌了：当AI不再需要代码编辑器》

详细风格分析和学习要点请参阅：`references/cursor-reference.md`

该文件包含：
- 完整原文
- 逐段风格分析（开头设计、数据使用、小标题、权威引用等）
- 10大核心写作手法总结
- 具体技巧拆解

**重要提示**：在开始改写前，建议快速浏览该参考文件，理解爆文的核心特征。改写时灵活运用这些技巧，而非机械模仿。

## 完整工作流程

### 第一步：读取并分析原文

1. **读取原文**
   - 使用Read工具读取用户指定的文档
   - 如果用户只给了文件名，先用Glob查找

2. **深度分析原文逻辑**（创建临时分析文件）
   提取以下要素：

   **核心论点**：文章的主要观点是什么？

   **逻辑框架**：
   - 论证结构（几层论述）
   - 每层的核心论据
   - 论证方式（演绎/归纳/类比）

   **优势**：
   - 理论深度如何
   - 逻辑是否严密
   - 有哪些好的类比或框架

   **劣势**（这是改写的关键）：
   - ❌ 缺乏开头冲击力
   - ❌ 缺乏具体数据和案例
   - ❌ 理论偏重，代入感不足
   - ❌ 段落过长，节奏慢
   - ❌ 缺乏冲突感和戏剧性
   - ❌ 标题不够吸引眼球

将分析结果保存到 `原文分析-大纲.md`

### 第二步：互联网搜索（数据挖掘）

基于原文主题，并行搜索5-7个方向，收集：

**必搜项**：
1. 最新数据（2025-2026）
   - 行业增长数据
   - 公司收入数据
   - 用户规模数据
   - 市场份额数据

2. 真实案例
   - 知名公司实践
   - 创业者故事
   - 具体项目成果
   - 转型案例

3. 权威观点
   - 行业专家引用
   - 公司CEO/创始人观点
   - 研究机构报告

**搜索策略**：
- 使用WebSearch工具并行搜索
- 每个搜索聚焦一个具体方向
- 关键词要包含年份（2025/2026）
- 优先搜索一手数据来源

**数据要求**：
- 每个论点至少有1-2个数据支撑
- 数据要有具体数字（不要"大幅增长"，要"增长40%"）
- 标注数据来源（用于文末引用）

### 第三步：编写爆文大纲

创建 `新大纲-爆文版.md`，包含：

**1. 标题方案（3选1）**
- 标题1：数字冲击型（"这个程序员2个月没写一行代码，却提交了1000+个PR"）
- 标题2：悬念反转型（"当100%的代码都由AI写：程序员的终极进化还是末日警钟？"）
- 标题3：权威数据型（"腾讯90%工程师都在用的秘密：他们不再是码农，而是AI工厂厂长"）

选择标准：**具体数字 + 悬念 + 用户关心的话题**

**2. 文章结构框架**

```
开篇：戏剧性场景（300-500字）
├─ 冲击性开头：具体人物 + 具体场景
├─ 引出矛盾：数据A vs 数据B
└─ 抛出核心问题

主体：3-5个部分（每部分600-900字）
├─ 每部分有小标题（疑问句或悬念句）
├─ 数据 + 案例 + 分析
└─ 短段落（不超过4行）

结尾：升华 + 互动（300字）
├─ 收尾金句
└─ 评论区钩子（问题/投票）
```

**3. 数据库**
列出所有收集的：
- 关键数据（带来源标注）
- 真实案例（公司/人物/结果）
- 权威引用（人物/观点）

**4. 写作风格要点**
- ✅ 开头3秒抓眼球
- ✅ 每个论点有数据
- ✅ 至少5个真实案例
- ✅ 加粗关键信息
- ✅ 段落不超过4行
- ✅ 制造冲突感
- ✅ 快-慢-快节奏

### 第四步：改写正文

按照大纲，创建新文件：`改写完成版-[新标题].md`

**开篇写作（最关键）**：

```markdown
# [冲击性标题]

[年份/月份]的一个[时间]。

[具体人物]在[平台]上发了一条让整个[圈子]炸锅的[内容]：

**"[一句话冲击性引用，必须有具体数字]"**

评论区瞬间分裂成两个阵营：
一半人[反应1]，另一半人[反应2]。

但数据不会骗人。

同一周/月，[数据1]显示：**[具体数字+结论]**。[数据2]显示：**[具体数字+结论]**。

可与此同时，[对比数据]却显示：**[矛盾现象]**。

**这个行业到底发生了什么？**
[核心问题]
```

**主体写作原则**：

1. **数据密集**
   - 每3-4段必有一个数据
   - 数据格式：**加粗数字**
   - 示例："年化收入突破**20亿美元**"

2. **案例故事化**
   ```markdown
   **案例N：[公司/人物的具体实践]**

   [具体时间/背景]，[公司/人物]通过[方法]，[具体结果]。

   [关键数据]显示：[具体成果]。

   [人物引用]："[一句话总结]"
   ```

3. **小标题吸引**
   - ❌ "工作方式的改变"
   - ✅ "一边是每天20个PR，一边是手写90%代码"

4. **段落控制**
   - 每段2-4行
   - 单句成段制造节奏
   - 重要信息独立成段

5. **冲突制造**
   ```markdown
   **旧模式**：你亲自在流水线上拧螺丝
   **新模式**：你设计流程，智能体执行生产
   ```

**结尾写作**：

```markdown
## [升华小标题]

[总结性论述，3-4段]

**编程作为主要是敲键盘活动的时代结束了。**

**编程作为主要是思考和判断活动的时代，刚刚换到更高档位。**

[核心隐喻/金句]

---

## 💬 [互动问题]

**A.** [选项1]
**B.** [选项2]
**C.** [选项3]

评论区留下你的选择和真实体验。

**转发给你的[目标读者]，一起变强！**

---

## 🔥 关注「[公众号名]」

[slogan]

---

## Sources / 数据来源

[1] [标题](URL)
[2] [标题](URL)
...
```

### 第五步：质量检查

改写完成后，自检以下要点：

- [ ] 开头3秒有冲击力吗？（具体人物+具体数字+悬念）
- [ ] 每个论点都有数据支撑了吗？
- [ ] 至少有5个真实案例了吗？
- [ ] 关键信息加粗了吗？
- [ ] 段落都短于5行了吗？
- [ ] 有制造冲突感吗？（新旧对比、数据对比）
- [ ] 节奏是否快-慢-快？
- [ ] 结尾有互动钩子吗？
- [ ] 所有数据来源都列出了吗？
- [ ] 标题够吸引人吗？

### 第六步：清理临时文件

询问用户是否需要保留分析文件：
- 如果不需要：删除 `原文分析-大纲.md` 和 `新大纲-爆文版.md`
- 如果需要：保留所有文件

**默认行为**：仅保留 `改写完成版-[标题].md`

## 爆文风格核心要素

### 1. 戏剧性开头（参考《Cursor慌了》）

**反面案例**：
> "最近AI编程工具发展很快，Cursor是其中的代表。"

**正面案例**：
> "2026年1月5日，Cursor的员工们结束假期回到公司，迎接他们的是一场全员大会，演示文稿的标题赫然写着**'战时状态'**。"

**要素**：
- 具体时间/地点
- 具体人物/公司
- 具体场景/事件
- 冲击性细节

### 2. 数据密集

**反面案例**：
> "很多工程师在使用AI工具，效率大幅提升。"

**正面案例**：
> "腾讯**90%**的工程师正在使用AI编码。Cursor的年化收入突破**20亿美元**，其中**60%**来自企业客户。Claude Code仅用6个月就把ARR从0做到了**25亿美元**。"

**原则**：
- 具体数字，不要"很多"、"大幅"
- 数字加粗
- 标注来源

### 3. 真实案例

**反面案例**：
> "有公司使用AI工具后效率提升了。"

**正面案例**：
> "**案例1：创业者刘小牌的魔法**
>
> 这位创业者通过Claude Code驱动AI并行开发数十个产品，将原本**15人团队的工作量压缩到2人完成**。
>
> 他每天的工作流程变成了：
> - 早上：给20个智能体下'任务单'
> - 中午：审查前一批次的产出
> - 下午：重新分配出错的任务
> - 晚上：合并通过验收的PR"

**要素**：
- 具体公司/人物
- 具体方法/流程
- 具体结果/数据
- 细节描写

### 4. 冲突制造

**对比方式**：
- 新旧对比
- 数据对比
- 阵营对比
- 预期vs现实

**示例**：
```markdown
调查揭示了一个惊人的**双峰分布**：

一边是像Boris Cherney这样的人——每天提交数十个PR，代码100%由AI编写。

另一边则是将近半数的开发者，手动编写的代码比例仍超过90%。

这两个群体就像生活在两个平行宇宙。
```

### 5. 短段落快节奏

**段落长度控制**：
- 开篇：1-2行一段
- 主体：2-4行一段
- 关键信息：单句成段

**示例**：
```markdown
评论区瞬间分裂成两个阵营：

一半人惊呼"程序员要失业了"，另一半人冷笑"又一个营销号在炒作"。

但数据不会骗人。

同一周，腾讯内部数据显示...
```

### 6. 权威背书

**引用方式**：
```markdown
软件架构大师Grady Booch把这个阶段称为**"软件的第三纪元"**——一个由抽象层级提升定义的新黄金时代。

Cursor团队说得很直白：**"开发者的工作正在变成构建生产软件的系统——工厂，而不只是产品本身。"**
```

**来源**：
- 行业专家
- 公司CEO/创始人
- 研究机构
- 权威媒体

## 常见问题

**Q: 如果原文缺乏数据怎么办？**
A: 必须通过WebSearch补充。每个论点至少要有1-2个数据支撑。如果实在找不到相关数据，考虑：
- 扩大搜索范围（相关行业、类似案例）
- 使用间接数据（上下游数据、竞品数据）
- 引用专家观点作为补充

**Q: 如果搜不到2025-2026的数据？**
A: 使用最新可得数据，并明确标注时间。示例："截至2024年底的数据显示..."

**Q: 原文很学术，改成爆文会不会降低专业度？**
A: 爆文 ≠ 降低专业度。核心是：
- 保留所有关键论点和逻辑
- 用更生动的方式呈现（数据、案例、故事）
- 降低阅读门槛，不降低思考深度

**Q: 多长的文章适合改写？**
A: 原文3000-10000字最佳。改写后通常会增加20-50%（因为增加了数据和案例）。

**Q: 需要改标题吗？**
A: 必须改。标题是爆文的第一要素。提供3个备选，让用户选择。

## 注意事项

1. **不要牺牲准确性**
   - 所有数据必须真实
   - 引用必须准确
   - 不要夸大或杜撰

2. **保留原文精华**
   - 核心论点不变
   - 逻辑框架不变
   - 深度见解不变

3. **适度戏剧化**
   - 可以增加戏剧性，但不要哗众取宠
   - 可以制造冲突，但不要误导读者
   - 可以用强烈措辞，但要基于事实

4. **尊重原作者**
   - 如果原文有作者信息，在改写版中保留
   - 如果是改写已发表文章，注明"改写自XXX"

5. **数据来源**
   - 所有数据必须标注来源
   - 文末必须列出完整的Sources列表
   - 使用markdown链接格式

## 工作流程总结

```
用户请求
    ↓
读取原文 (Read)
    ↓
分析逻辑 (Write 分析文件)
    ↓
搜索数据 (WebSearch × 5-7)
    ↓
编写大纲 (Write 大纲文件)
    ↓
改写正文 (Write 最终文件)
    ↓
质量检查 (自检清单)
    ↓
清理文件 (Bash rm 或保留)
    ↓
交付成果
```

**关键点**：
- 分析要深入（找出原文所有劣势）
- 搜索要广泛（5-7个并行搜索）
- 大纲要详细（包含数据库和风格要点）
- 改写要用心（每个要素都到位）
- 检查要严格（质量清单逐项核对）
